## 1. 제품 개요

### 1.1 제품명
- **KR Stock Daily Brief**

### 1.2 한 줄 설명
- 한국 주식 시장 데이터를 바탕으로 매일 요약을 생성하고, 달력 UI에서 날짜별로 조회/누적 관리하는 내부용 웹앱.

### 1.3 목표
- 평일 장 마감 이후, 요약이 자동 생성되어 달력에 누적 표시되도록 한다.
- 수동 생성/조회도 언제든 가능하도록 한다.
- Docker 기반으로 어디서든 동일하게 실행되도록 한다.

## 2. 문제 정의

현재 사용자는 다음 문제를 겪는다:
- 매일 시장 핵심 포인트(상승/하락/관심 종목)를 수동으로 확인해야 함
- 과거 날짜 요약을 한 곳에서 누적/조회하기 어려움
- 개발/운영 환경마다 실행 방식이 달라 관리 부담이 큼

해결 방향:
- 자동 생성 + 달력 기반 조회 + 표준화된 Docker 실행 환경 제공

## 5. 핵심 기능 요구사항

### FR-1. 요약 생성
- 사용자는 특정 날짜 요약을 생성할 수 있어야 한다.
- 시스템은 생성 결과를 DB에 저장해야 한다.
- **이미 존재하는 날짜를 다시 생성(리제너레이트/덮어쓰기)하는 기능은 관리자만 가능**해야 한다.
- 오늘 날짜 생성 API를 제공해야 한다.

### FR-2. 요약 조회
- 사용자는 기간 조회(from~to)로 월 단위 요약 목록을 조회할 수 있어야 한다.
- 사용자는 날짜별 단건 조회를 할 수 있어야 한다.

### FR-3. 달력 UI
- 월 달력에서 날짜를 선택하면 해당 날짜 상세 요약을 보여줘야 한다.
- 데이터가 있는 날짜는 dot 표시로 시각화해야 한다.
- 생성 직후 dot 표시가 즉시 갱신되어야 한다.

### FR-4. 자동 생성 스케줄
- 평일 15:40 Asia/Seoul 기준 자동 생성이 수행되어야 한다.
- 생성 성공/실패 상태를 리포트할 수 있어야 한다.

### FR-5. 접근 게이트(옵션)
- `PUBLIC_KEY`가 설정되면 API/UI 접근에 `k` 파라미터 검증을 적용해야 한다.

### FR-6. 관리자 인증 및 신뢰 CIDR 우회
- `ADMIN_KEY`가 설정되면 관리자 전용 작업(재생성, 백필, 보관)에 인증이 필요하다.
- 인증 방식:
  - 키 기반: `X-Admin-Key` 헤더 또는 `adminKey` 쿼리 파라미터
  - 신뢰 CIDR 기반: 요청 IP가 `APP_ADMIN_TRUSTED_CIDRS`에 포함되면 키 없이 관리자 권한 부여
- 기본 신뢰 CIDR: `127.0.0.1/32,::1/128` (localhost만)
- **프로덕션 배포 시 주의사항**:
  - 인터넷 공개 배포 시 `APP_ADMIN_TRUSTED_CIDRS=""`로 설정하여 우회 비활성화 권장
  - 리버스 프록시 사용 시 `X-Forwarded-For` 헤더 검증 필요
  - Docker 내부 네트워크 사용 시 `172.16.0.0/12` 등 적절한 CIDR 추가 고려

### FR-7. 데이터 누적
- 날짜 단위 누적 저장되어야 한다.
- 동일 날짜 재생성(업데이트/덮어쓰기)은 **관리자만** 수행할 수 있어야 한다.
- 일반 사용자는 이미 존재하는 날짜에 대해 재생성을 시도할 수 없다(정책적으로 차단).

### FR-8. 요약 보관(삭제 대체)
- 사용자는 특정 날짜 요약을 보관(archive) 처리할 수 있어야 한다.
- 보관 후 달력 dot/통계/인사이트가 즉시 반영되어야 한다.
- 물리 삭제 대신 보관 방식으로 복구 가능성을 유지해야 한다.

### FR-9. 과거 데이터 불러오기(백필)
- 관리자는 기간 지정으로 과거 날짜를 일괄 생성/보강할 수 있어야 한다.
- 일반 사용자는 백필을 수행할 수 없다.
- 백필 결과는 날짜별 성공/실패/원인으로 리포트되어야 한다.
- 데이터 소스 제약(정확도/호출 제한)을 명시적으로 표시해야 한다.

## 7. 데이터 소스 정책 (현재)

### 7.1 기본 소스
- 과거 날짜는 `pykrx` sidecar(`/leaders`)를 우선 사용한다.
- pykrx 실패 시 `MARKETDATA_PROVIDER`(기본 naver)로 fallback한다.

### 7.2 현재 규칙
- topGainer/topLoser: `get_market_price_change_by_ticker(직전영업일, 대상일, ALL)` 기준
- mostMentioned: 거래량 최대 종목(대체 규칙)
- KOSPI/KOSDAQ pick: 각 시장 거래량 최대 종목(대체 규칙)
- 큰 등락률만으로 자동 제외하지 않는다(수치 유지 + 설명 제공)

### 7.3 Generate 기능 정의
- Generate는 "요약을 새로 생성하거나(없으면 insert) 다시 계산해 갱신하는(upsert)" 기능이다.
- 입력 날짜(또는 오늘, Asia/Seoul 기준)에 대해 데이터 소스에서 요약 후보를 가져와 저장한다.
- 과거 날짜는 pykrx 기반 참조를 우선 사용해 일관된 검증/설명 데이터를 유지한다.

### 7.4 rawNotes 정의
- `rawNotes`는 요약 생성의 근거/메타 정보를 담는 필드다.
- 예: `Source: pykrx`, 규칙 버전, fallback 사유, 데이터 소스 실패 이유.
- 사용자에게 디버깅 및 신뢰도 판단 근거를 제공한다.

### 7.5 설명/투명성 필드
- API는 아래 투명성 필드를 제공한다.
  - `rawTopGainer/rawTopLoser`: 처음 계산된 순위
  - `filteredTopGainer/filteredTopLoser`: 검토 후 최종 표시 순위
  - `anomalies[]`, `rankingWarning`: 주의 신호와 안내
  - `leaderExplanations`: 초보자용 한 줄 설명 + 근거 링크
- 설명은 규칙 기반(비-LLM)으로 생성하며, 단정 표현 대신 근거 중심 문구를 사용한다.

### 7.6 failed fetch 정의
- failed fetch는 외부 데이터 소스 호출 실패(응답 없음, 파싱 실패, 네트워크 오류 등)를 의미한다.
- 시스템은 재시도 후 실패 원인을 `rawNotes` 및 로그로 남긴다.
- API는 가능한 경우 정상 응답을 유지하되, 데이터 품질 저하 상태임을 명시한다.

### 7.7 과거 데이터(백필) 확보 전략
- 1안(현재 우선): 날짜 루프를 돌며 v1 소스로 백필 시도, 실패일은 리포트에 누락/원인 표시
- 2안(정확도 강화): pykrx/KRX/기타 공식 소스 연계(라이선스/호출정책 확인 필요)
- 3안(혼합): 최근 구간은 v1, 특정 검증 구간은 보강 소스로 재생성

참고 조사(웹/문서):
- pykrx GitHub 문서에서 KRX/Naver 스크래핑 기반 라이브러리 성격 확인
- 공개 웹 검색 API 키 제한으로 일부 자동 탐색은 제약이 있어, 구현 시 소스별 접근성 검증을 병행

## 8. API 요약

현재 구현:
- `GET /api/summaries/stats`
- `GET /api/summaries/insights?from=YYYY-MM-DD&to=YYYY-MM-DD`
- `GET /api/summaries?from=YYYY-MM-DD&to=YYYY-MM-DD`
- `GET /api/summaries/latest`
- `GET /api/summaries/{date}`
- `POST /api/summaries/{date}/generate`
- `PUT /api/summaries/{date}/archive` (요약 보관)
- `POST /api/summaries/backfill?from=YYYY-MM-DD&to=YYYY-MM-DD` (과거 일괄 생성)
- `POST /api/summaries/generate/today`

(상세는 `docs/API_SPEC.md` 참조)
