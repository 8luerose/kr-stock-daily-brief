## 0. 이번 목표

이번 목표는 `프론트 전면 삭제 수준의 재설계와 재구현`이다.

기존 프론트 화면, 레이아웃, CSS, 컴포넌트 구조에 얽매이지 마라.

단, 백엔드 기능과 기존 API는 삭제하지 마라.

테스트, 감사, Docker, E2E, make quality, 전체 API 검증부터 하지 마라.
이번 작업의 1순위는 코딩이다.
먼저 눈에 보이는 프론트 화면을 완전히 바꿔라.

## 1. 최종 제품 방향

내가 원하는 서비스는 버튼이 적고 깔끔하면서, AI가 한국 주식을 초보자에게 쉽게 설명하고 예측해주는 주식 학습 서비스다.

최종 제품 한 줄:

> 초보자가 차트 위 AI 주석을 보며 왜 올랐는지, 왜 내렸는지, 언제 매수와 매도를 검토할 수 있는지 배우는 한국 주식 AI 학습/리서치 플랫폼

사용자가 첫 화면에서 바로 느껴야 하는 것:

- 버튼이 적고 깔끔하다.
- 무엇을 검색하면 되는지 바로 보인다.
- 그래프가 메인이다.
- AI가 차트 위에서 하나하나 설명한다.
- 언제 사고 언제 팔아야 하는지, 어떤 조건에서 검토해야 하는지 보인다.
- 호재와 악재가 무엇인지 보인다.
- 초보자도 주식 용어와 개념을 이해할 수 있다.
- 밋밋하지 않고, 애니메이션과 깊이감이 있다.

너는 한 명이지만 아래 역할을 모두 수행한다고 가정한다.

- Toss에서 10년차 이상, 연봉 1억 이상 받는 수준의 프론트엔드 개발자
- Toss에서 10년차 이상, 연봉 1억 이상 받는 수준의 백엔드 개발자
- Toss에서 10년차 이상, 연봉 1억 이상 받는 수준의 DevOps 개발자
- Toss에서 10년차 이상, 연봉 1억 이상 받는 수준의 AI 엔지니어
- 사용자가 첫 화면에서 바로 가치를 이해하게 만드는 제품 디자이너
- 차트 기반 투자 학습 UX를 설계하는 핀테크 UX 전문가
- 이 사업에 투자할지 판단하는 벤처캐피탈 투자자의 눈으로 제품성을 보는 감독관

이번 결과물은 단순히 동작하는 화면이면 안 된다.
아래 사람들이 봐도 납득할 수 있어야 한다.

1. 실제 사용자
   - 버튼이 적고 헷갈리지 않아야 한다.
   - 첫 화면에서 무엇을 하면 되는지 바로 알아야 한다.
   - 차트를 보며 왜 올랐고 왜 내렸는지 이해해야 한다.
   - 매수 검토, 관망, 매도 검토, 리스크 관리 시점을 조건형으로 이해해야 한다.
   - 초보자 용어 설명이 충분해야 한다.

2. 프론트엔드 개발자
   - 컴포넌트 구조가 작고 재사용 가능해야 한다.
   - App shell, Search, Chart, AI panel, Learning, Admin이 역할별로 분리되어야 한다.
   - CSS/design token 구조가 명확해야 한다.
   - PC/mobile 반응형, clipping, tooltip, animation, loading, empty, error state가 관리 가능해야 한다.
   - 접근성과 성능을 고려해야 한다.

3. 백엔드 개발자
   - 기존 API를 삭제하지 않아야 한다.
   - 프론트가 필요한 검색, 차트, 이벤트, AI, 학습, 관리자 API 계약이 명확해야 한다.
   - 데이터 기준일, 출처, 신뢰도, 한계가 응답 구조에 연결되어야 한다.
   - mock이나 fallback을 쓰면 코드와 화면에서 명확히 표시해야 한다.

4. DevOps 개발자
   - Docker, env, healthcheck, secret 관리, 배포 가능성을 해치면 안 된다.
   - `.env`와 secret은 절대 commit하지 않는다.
   - 테스트는 마지막에 하되, 마지막에는 build와 운영 위험을 확인할 수 있어야 한다.

5. AI 엔지니어
   - AI가 제품 핵심으로 보여야 한다.
   - 내부 사고 과정은 노출하지 않는다.
   - 답변은 결론, 근거, 반대 신호, 리스크, 기준일, 신뢰도, 한계, 출처 중심이어야 한다.
   - RAG/Agentic 구조가 화면과 코드에서 확장 가능하게 보여야 한다.

6. 투자자
   - 첫 화면에서 AI 주식 학습/리서치 플랫폼이라는 정체성이 보여야 한다.
   - 차트 위 AI 주석, 매수/매도 검토 시점, 호재/악재 요약이 차별화 포인트로 보여야 한다.
   - 사용자가 계속 돌아올 학습/관찰/포트폴리오 확장 가능성이 보여야 한다.
   - 단순 일간 브리프 도구가 아니라 확장 가능한 서비스처럼 보여야 한다.

## 8. UI/UX 감성

- Toss처럼 단정하고 신뢰감 있어야 한다.
- 심플하지만 심심하지 않아야 한다.
- 버튼은 모바일 앱처럼 적고 명확해야 한다.
- Pretendard를 한국어/영어 기본 폰트로 적용한다.
- 데이터, 차트, 검색, AI 결과가 주인공이 되게 한다.
- hover, focus, active, panel transition, tooltip, loading, empty, error state에 150~320ms 수준의 부드러운 micro interaction을 넣어라.
- 글래스모피즘은 과하지 않게 사용한다.
- 밋밋한 2D 화면처럼 보이면 실패다.
- 과한 장식으로 덮지 말고, 차트와 AI 주석이 살아있게 만들어라.

추가 품질 기준:

- 빈 상태도 예뻐야 한다.
- 로딩 상태도 예뻐야 한다.
- 에러 상태도 사용자가 다음 행동을 알 수 있어야 한다.
- 버튼, 탭, 차트 마커, AI 패널, 학습 카드에 hover/focus/active 상태가 있어야 한다.
- 키보드 접근성과 터치 영역을 고려한다.
- 모바일 390px에서 텍스트, 버튼, tooltip, 차트가 겹치지 않아야 한다.
- PC 1440px에서 화면 끝에 딱 붙거나 잘리는 요소가 없어야 한다.
- 애니메이션은 있어야 하지만, 과하게 느리거나 산만하면 안 된다.
- 색상, 여백, 폰트, 그림자, border radius는 디자인 토큰으로 관리한다.

## 8.5 참고 서비스 리서치

점수가 부족하거나 화면이 여전히 마음에 들지 않으면 구현을 멈추는 것이 아니라, 짧게 참고 서비스를 보고 바로 다시 구현한다.

참고 대상:

- Toss
- 고품질 증권 앱
- 고품질 핀테크 서비스
- AI SaaS 서비스
- 데이터 대시보드
- 모바일 앱 수준 웹 UX

참고할 항목:

- 첫 화면에서 사용자가 무엇을 해야 하는지 알 수 있는가
- 버튼 수가 얼마나 적고 명확한가
- 차트와 데이터가 얼마나 주인공처럼 보이는가
- AI 기능이 어디에 있고 왜 유용한지 바로 보이는가
- 모바일에서 답답하지 않은가
- 빈 상태, 로딩 상태, 에러 상태가 자연스러운가
- 애니메이션이 고급스럽고 빠른가

주의:

- 리서치에 오래 머물지 마라.
- 참고 후 바로 구현으로 돌아간다.

## 20. Git 커밋 메시지 작성 규칙

커밋 전에는 반드시 최근 커밋을 확인한다.

```bash
git log --oneline -10
```

기존 프로젝트의 커밋 메시지 흐름을 확인한 뒤, 아래 형식을 따른다.

```text
[FEAT]: (1) 내용1 / (2) 내용2 / (3) 내용3
```

대괄호 안에는 커밋의 성격을 쓴다.

- `[FEAT]`: 새로운 기능을 만들었을 때
- `[REFACTOR]`: 기존 기능의 구조나 동작 방식을 변경했을 때
- `[CHORE]`: 자잘한 코드, 설정, 정리 작업을 했을 때
- `[DOCS]`: 새 문서를 작성하거나 기존 문서를 수정했을 때
- `[STYLE]`: 기능 변화 없이 코드 스타일, UI 스타일, 포맷을 정리했을 때

커밋 메시지는 어려운 말이나 줄임말을 피한다.
작성자가 아닌 다른 사람이 봐도 바로 이해할 수 있어야 한다.

커밋하기 전에는 해당 커밋에 적합한 설명 후보를 `(N)`개로 나열하고, 그 설명을 포함한 커밋 메시지 예시를 제시한다.
상황상 직접 커밋해야 한다면 가장 명확한 메시지를 선택한다.

예시:

```text
[FEAT]: (1) 첫 화면을 AI 예측 차트 중심으로 다시 구성 / (2) 매수와 매도 검토 구간을 차트 위에 표시 / (3) 호재와 악재 요약 패널을 추가
```
