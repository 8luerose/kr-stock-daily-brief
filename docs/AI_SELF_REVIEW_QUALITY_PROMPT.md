5. 하네스 엔지니어링 원칙

모든 AI 관련 기능과 평가 과정은 즉흥적으로 처리하지 말고 하네스 절차를 따라라.

하네스 절차:

1. Intent 이해
- 사용자가 무엇을 원하는지 분류한다.
- 예: 시장 요약, 종목 분석, 차트 분석, 용어 설명, 매수/매도 검토, 포트폴리오 점검

2. Data 확인
- 필요한 데이터가 실제로 있는지 확인한다.
- 없는 데이터는 만들지 말고 "없음" 또는 "추후 필요"라고 표시한다.

3. Draft 생성
- 근거 기반으로 초안을 만든다.

4. Self Review
- 답변 또는 기능이 목적에 맞는지 스스로 검토한다.

5. Risk Review
- 과장, 단정, 출처 없는 주장, 위험한 투자 표현을 제거한다.

6. UX Review
- 사용자가 이해하기 쉬운지, 화면에서 자연스럽게 보이는지 검토한다.

7. Test Review
- 실제 API, 버튼, 화면, 브라우저, 빌드 결과로 확인한다.

8. Finalize
- 검증된 내용만 최종 반영한다.

중요:
- 내부 사고 과정은 사용자에게 길게 노출하지 않는다.
- 대신 결과, 점수, 개선 내용, 테스트 결과는 명확하게 보고한다.

6. 프론트엔드 최상급 품질 요구

프론트는 이 프로젝트의 핵심이다. 최대한 공을 들여라.

반드시 달성할 것:
- 최신 모던 SaaS/핀테크 스타일
- Toss처럼 단정하고 신뢰감 있는 UI
- 그래프가 메인인 화면
- 일봉/주봉/월봉 전환
- 차트 마커와 툴팁
- 매수 검토 구간/매도 검토 구간/관망 구간/리스크 구간 시각화
- 부드러운 애니메이션
- 빠른 반응성
- 모바일 완성도
- 예쁜 빈 상태/로딩 상태/에러 상태
- 초보자가 바로 이해하는 문구

주의:
- 카드만 많이 늘어놓지 마라.
- 텍스트가 겹치지 않게 하라.
- 버튼은 의미가 즉시 이해되어야 한다.
- 차트 툴팁은 화면 밖으로 나가지 않게 하라.
- 애니메이션은 과하지 않게, 고급스럽게 넣어라.
- 사용자가 "이제 뭘 눌러야 하지?"라고 느끼면 실패다.

7. 그래프와 매수/매도 판단 UX

사용자는 그래프를 보고 "언제 사고 팔아야 하는지"를 이해하고 싶어한다.

구현 방향:
- 직접적인 수익 보장이나 무조건적 투자 지시는 피한다.
- 대신 차트 기반의 "검토 구간"으로 보여준다.

표현 예시:
- 매수 검토 구간
- 분할매수 검토 구간
- 관망 구간
- 매도 검토 구간
- 손절/리스크 관리 구간

설명 예시:
- "20일선 회복과 거래량 증가가 함께 나오면 매수 검토 구간으로 볼 수 있습니다."
- "급등 후 거래량 감소와 긴 윗꼬리가 반복되면 일부 매도 검토 구간으로 볼 수 있습니다."
- "전저점 이탈 시 리스크 관리 기준을 다시 세워야 합니다."

차트에 포함할 것:
- 일봉
- 주봉
- 월봉
- 거래량
- 이동평균선
- 이벤트 마커
- 매수/매도 검토 구간
- 호버 툴팁
- AI 설명 패널
