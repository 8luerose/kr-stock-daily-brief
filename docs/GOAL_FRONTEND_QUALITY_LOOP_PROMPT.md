제품 감성 목표:
- Toss처럼 사소한 UI/UX까지 퀄리티가 있어야 한다.
- 단정하고 신뢰감 있으며, 심플하지만 절대 심심하지 않아야 한다.
- 버튼 수는 모바일 앱처럼 적고 명확해야 한다.
- 주요 액션은 1~3개만 전면에 보여라.
- 나머지 기능은 탭, 바텀시트, 오버플로 메뉴, 접힌 관리자 패널, 검색 중심 UX로 정리하라.

  - 얇은 border

- 그래프와 데이터가 주인공이어야 한다.
- AI 기능은 숨기지 말고 명확히 보여라.
- 한국어와 영어 모두 Pretendard를 기본 폰트로 통일하라.

프론트 전면 재설계 요구:

1. 첫 화면 재설계
- 첫 화면에서 서비스 정체성이 즉시 보여야 한다.
- "오늘 한국 주식시장을 AI와 차트로 이해하는 플랫폼"이라는 느낌이 나야 한다.
- 달력 중심, 운영자 중심 화면에서 벗어나라.
- 첫 화면의 핵심 구성:
  - 오늘 시장 핵심 요약
  - 산업/테마/기업/종목 통합 검색
  - 메인 차트 또는 대표 종목 차트
  - AI 분석 패널
  - 상승/하락/언급 TOP 종목
  - 배우기/용어/차트 해석 진입점
- 사용자는 첫 화면에서 "검색하거나, 차트를 보거나, AI에게 물어보면 된다"를 바로 알아야 한다.

2. 버튼 수 대폭 축소
- 현재 버튼이 많아 헷갈리는 문제를 최우선으로 해결하라.
- 전면 노출 버튼은 정말 중요한 것만 남긴다.
- 생성, 백필, 보관 같은 운영 버튼은 일반 사용자에게 보이지 않게 하라.
- 관리자 기능은 접힌 관리자 패널, 개발자 모드, URL key 기반 admin 영역으로 분리하라.
- 버튼 대신 더 적절한 UI를 써라.
  - 검색
  - 탭
  - segmented control
  - dropdown
  - toggle
  - bottom sheet
  - contextual action
- 같은 의미의 버튼이 여러 곳에 반복되면 제거하라.

3. 검색 기능 강화
- 사용자가 원하는 산업, 테마, 기업, 종목을 검색할 수 있어야 한다.
- 검색창은 첫 화면의 핵심 인터랙션이어야 한다.
- 검색 대상:
  - 기업명
  - 종목 코드
  - 산업
  - 테마
  - 시장 구분(KOSPI/KOSDAQ)
  - 용어
  - 오늘 움직인 종목
- 검색 결과는 단순 리스트가 아니라 다음 정보를 보여야 한다.
  - 종목명/코드
  - 시장
  - 최근 등락률
  - 관련 테마/산업
  - AI가 이 종목을 설명할 수 있다는 진입점
  - 차트 보기 진입점
- 실제 데이터가 부족하면 mock 또는 기존 top lists를 기반으로 UX를 먼저 완성하라.
- 단, mock임을 코드/문서에 명확히 표시하고 추후 API 연결 지점을 남겨라.

4. AI 기능을 명확히 보이게 만들기
- 벤처캐피탈 투자자가 봐도 AI 기능이 어디인지 보여야 한다.
- AI 기능을 작은 버튼 하나에 숨기지 마라.
- AI 패널은 제품의 핵심 가치처럼 보여야 한다.
- 예:
  - "AI 시장 해석"
  - "AI 차트 판단"
  - "AI 리스크 요약"
  - "AI에게 이 종목 물어보기"
  - "왜 올랐는지 AI가 근거로 설명"
- AI 답변은 다음 구조로 보여라.
  - 한 줄 결론
  - 근거
  - 반대 신호
  - 리스크
  - 출처
  - 신뢰도
  - 데이터 기준일
- "AI가 실제로 유용하다"는 느낌이 나야 한다.

5. 그래프/차트 중심 UX
- 그래프는 페이지의 메인이다.
- 일봉, 주봉, 월봉 전환이 자연스럽고 명확해야 한다.
- 차트가 비어 보이면 실패다.
- 차트에는 다음 요소가 있어야 한다.
  - 가격 흐름
  - 거래량
  - 이동평균선
  - 상승/하락 이벤트 마커
  - 공시/뉴스/언급량 마커
  - 매수 검토 구간
  - 분할매수 검토 구간
  - 관망 구간
  - 매도 검토 구간
  - 손절/리스크 관리 구간
- 마커 hover 시 AI 설명 툴팁이 떠야 한다.
- 툴팁은 화면 밖으로 나가면 안 된다.
- 모바일에서도 차트 조작이 답답하면 안 된다.

6. 매수/매도 판단 UX
- 사용자는 그래프를 보고 언제 사고 팔아야 하는지 이해하고 싶어한다.
- 직접적인 수익 보장이나 무조건적 투자 지시는 피하되, 차트 기반 판단 경험은 선명하게 제공하라.
- 표현은 단정 대신 검토/조건/시나리오 방식으로 한다.
- 예:
  - "매수 검토 구간"
  - "분할매수 검토 구간"
  - "관망 구간"
  - "매도 검토 구간"
  - "리스크 관리 구간"
- 각 구간에는 아래를 반드시 표시한다.
  - 조건
  - 근거
  - 반대 신호
  - 신뢰도
  - 데이터 기준일
  - 초보자 설명
- 예시 문구:
  - "20일선 회복과 거래량 증가가 함께 확인되면 매수 검토 구간으로 볼 수 있습니다."
  - "급등 후 거래량 감소와 긴 윗꼬리가 반복되면 일부 매도 검토 구간으로 볼 수 있습니다."
  - "전저점 이탈 시 리스크 관리 기준을 다시 세워야 합니다."

7. 배우기 탭 전면 강화
- 배우기 탭은 부가 기능이 아니라 초보자 retention 기능이다.
- 각 용어는 다음 구조를 가져야 한다.
  - 첫 줄 핵심 요약 1문장
  - 최소 3줄 이상의 쉬운 설명
  - 왜 중요한가
  - 차트에서 언제 확인하는가
  - 초보자가 자주 하는 오해
  - 실제 시나리오 예시
  - 관련 질문
- "등락률", "거래량", "거래대금", "PER", "PBR", "ROE", "공시", "DART", "일봉", "월봉", "이동평균선", "손절", "분할매수", "추세", "저항선", "지지선" 같은 용어를 우선 강화하라.
- 텍스트는 길어도 읽기 쉽게 접힘/요약/예시 구조로 설계하라.

8. 애니메이션과 depth
- 현재 화면이 너무 2D처럼 느껴지는 문제를 해결하라.
- 단, 과한 효과는 금지한다.
- 넣어야 할 micro-interaction:
  - 탭 전환
  - 검색 포커스
  - 카드 hover
  - 차트 마커 hover
  - AI 패널 등장
  - 학습 항목 expand/collapse
  - 로딩 skeleton
  - empty state motion
  - 버튼 press
- 애니메이션은 150ms~320ms 범위에서 빠르고 고급스럽게 처리하라.
- 사용자 설정의 reduced motion도 고려하라.

9. Pretendard 폰트
- 한국어, 영어, 숫자 모두 Pretendard를 우선 사용한다.
- 폰트 fallback을 둔다.
- font-size, line-height, font-weight를 재정리하라.
- 좁은 화면에서 글자가 답답하거나 겹치면 실패다.

하네스 엔지니어링 절차:
모든 개선 루프는 아래 절차를 따른다.

1. Intent Understanding
   - 사용자의 불만을 다시 정리한다.
   - 핵심은 "버튼 과다", "복잡함", "Toss스럽지 않음", "AI 기능이 안 보임", "검색 부재", "프론트 전면 재설계"다.

2. Current State Audit
   - 실제 코드를 읽고 현재 화면 구조를 파악한다.
   - 로컬 서버를 실행하고 브라우저에서 직접 본다.
   - 추측하지 말고 실제 화면, 코드, API 기준으로 판단한다.

3. Multi-Perspective Scoring
   - 사용자 관점 500점
   - 개발자 관점 500점
   - 벤처캐피탈/주주 관점 500점
   - 각 관점은 5개 항목 x 100점으로 평가한다.

4. Gap Analysis
   - 495점 이상이 아닌 관점의 부족한 점을 구체적으로 찾는다.
   - "왜 Toss스럽지 않은지", "왜 AI 기능이 안 보이는지", "왜 복잡한지"를 화면 기준으로 설명한다.

5. Reference Research
   - 495점 이상을 넘지 못하면 잘 만들어진 사이트 5곳 이상을 실제로 찾아본다.
   - 참고 대상:
     - Toss
     - 고품질 핀테크 서비스
     - 투자/증권 앱
     - AI SaaS
     - 데이터 대시보드
     - 모바일 앱 수준의 웹 UX
   - 각 사이트의 UI/UX, IA, 버튼 수, 검색 경험, AI 기능 노출, 애니메이션, 모바일 반응형, 에러/빈 상태를 분석한다.

6. Re-architecture Plan
   - 부족한 점과 참고 분석을 바탕으로 전체 프론트 구조를 다시 설계한다.
   - 필요하면 App.jsx와 styles.css를 전면 재구성한다.
   - 주요 컴포넌트를 분리하는 것도 허용한다.
   - 단, 변경 후 빌드와 테스트가 반드시 통과해야 한다.

7. Implement
   - 계획에 따라 작은 단위로 구현한다.
   - 버튼을 줄이고 검색 중심으로 바꾼다.
   - AI 기능을 명확하게 노출한다.
   - 차트와 학습 경험을 강화한다.

8. Test
   - 기능/버튼별로 직접 테스트한다.
   - 모바일/태블릿/데스크톱을 확인한다.
   - 빌드와 API 스모크 테스트를 실행한다.
   - 브라우저에서 직접 보고 "AI 템플릿 같다"는 느낌이 사라졌는지 확인한다.

9. Re-score
   - 다시 사용자/개발자/벤처캐피탈 관점으로 점수를 매긴다.
   - 각 관점이 495/500 이상인지 확인한다.

10. Loop or Report
   - 모든 관점이 495점 이상이면 최종 보고한다.
   - 하나라도 495점 미만이면 다시 Reference Research 또는 Re-architecture Plan 단계로 돌아간다.
