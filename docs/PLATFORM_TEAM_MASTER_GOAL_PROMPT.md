## 0. 이 문서의 목적

이 문서는 `kr-stock-daily-brief`를 단순한 한국 주식 일간 브리프 도구가 아니라, 프론트엔드·백엔드·DevOps·AI 개발팀이 함께 만드는 `출시 가능한 한국 주식 AI 리서치 웹 플랫폼`으로 끌어올리기 위한 종합 지시서다.

`/goal`에는 이 문서를 읽으라는 짧은 지시만 넣고, 실제 구현 기준은 이 문서를 따른다.

## 1. 최종 제품 목표

최종 제품 한 줄:

> 오늘 한국 주식시장을 AI, 검색, 차트, 초보자 설명, 포트폴리오 샌드박스로 이해하는 프리미엄 웹 플랫폼

사용자는 앱을 열자마자 아래를 이해해야 한다.

- 오늘 시장에서 어떤 종목이 움직였는가
- 왜 움직였는가
- 관련 산업, 테마, 기업, 종목을 어떻게 검색하는가
- 차트에서 어떤 구간을 봐야 하는가
- AI가 어떤 근거로 해석하는가
- 초보자가 무엇을 배워야 하는가
- 내 관심 종목/가상 포트폴리오를 어떻게 점검할 수 있는가

## 2. 현재 문제 인식

기존 개발 결과는 목표에 비해 부족하다.

- 프론트가 전면 재설계되지 않았고 기존 화면 위에 덧칠한 느낌이 강하다.
- PC 화면에서 `AI 인사이트`/검색 결과 하단이 잘린다.
- 헤더 문구가 좌측에 붙어 보인다.
- 버튼이 많고 복잡하다.
- Toss처럼 단정하고 고급스럽지 않다.
- 애니메이션과 depth가 부족하다.
- 그래프가 메인처럼 느껴지지 않는다.
- AI 기능이 제품의 핵심으로 명확히 드러나지 않는다.
- VC 투자자 관점에서 AI/RAG/Agentic 플랫폼 확장성이 약하게 보인다.
- 문서에는 `완성 단계`라고 되어 있지만 상세 체크리스트에는 미완료 항목이 많이 남아 있다.

따라서 다음 작업은 단순 수정이 아니라 제품·프론트·백엔드·AI·DevOps 전체를 다시 정렬하는 작업이다.

## 4. 팀별 역할

### 4.1 제품/디자인 팀

목표:

- 사용자가 5초 안에 서비스 가치를 이해하게 만든다.
- 첫 화면을 운영자 도구가 아니라 AI 주식 리서치 플랫폼으로 만든다.
- 버튼을 줄이고 검색, 차트, AI를 중심 경험으로 만든다.

해야 할 일:

- 전체 IA 재설계
- 첫 화면 command center 설계
- 검색/차트/AI/배우기/포트폴리오/관리자 영역 분리
- PC와 모바일 각각의 핵심 사용 흐름 정의
- Toss급으로 단정하고 신뢰감 있는 UX 설계

### 4.2 프론트엔드 팀

목표:

- 기존 화면 위에 덧칠하지 말고, 필요한 경우 프론트 구조를 전면 재설계한다.
- PC와 모바일에서 잘림, 겹침, 끝 붙음, 버튼 과다, 촌스러움이 없어야 한다.

필수 구현:

- App shell, header, main layout, navigation 재설계
- 전체 max-width, gutter, spacing scale 재정의
- PC `AI 인사이트`/검색 결과 clipping 해결
- 헤더 좌측 붙음 해결
- 통합 검색 command palette 또는 dedicated result panel 구현
- 메인 차트 workspace 구현
- 일봉/주봉/월봉 전환
- 차트 이벤트 마커와 tooltip
- 매수 검토, 분할매수 검토, 관망, 매도 검토, 리스크 관리 구간 표시
- AI 시장 해석, AI 차트 판단, AI 리스크 요약 패널 구현
- 배우기 탭 용어 설명 강화
- 포트폴리오 샌드박스 UX 정리
- 관리자 기능은 일반 사용자 화면에서 분리
- Pretendard 폰트 적용
- 150~320ms 범위의 절제된 micro-interaction 적용
- reduced motion 고려

품질 기준:

- Toss에서 10년차 이상, 연봉 1억 이상을 받는 프론트 개발자가 만든 것 같은 완성도
- 컴포넌트는 작고 재사용 가능해야 한다.
- hooks, API client, constants, copy, styles 책임을 분리한다.
- 유지보수성, 재사용성, 가독성을 최우선으로 둔다.

### 4.3 백엔드 팀

목표:

- 프론트 경험을 뒷받침하는 안정적인 API gateway 역할을 한다.
- 기존 일간 브리프, 요약 생성, 백필, 보관, 검증 기능을 깨지 않는다.

필수 구현/점검:

- 기존 summaries API 보존
- 검색 API 계약 정리
- 종목 상세 API 설계
- 차트 OHLCV API 설계
- 이벤트/마커 API 설계
- trade-zone API 설계
- AI service 연동 API 설계
- 포트폴리오 샌드박스 API 설계
- admin API와 일반 사용자 API 분리
- API_SPEC 문서 동기화

데이터 정확도 규칙:

- pykrx의 `get_market_price_change_by_ticker()` 등락률을 그대로 믿지 않는다.
- 전일 종가 대비 등락률은 OHLCV 당일/전일 종가로 직접 계산한다.
- 전일 또는 당일 거래량 0인 종목, 비정상 종목코드, 거래정지 의심 종목은 필터링한다.
- `0011T0` 같은 비6자리 숫자 코드는 제외한다.
- KOSPI/KOSDAQ 각각 상승/하락 1위와 TOP3 첫 번째 항목은 반드시 일치해야 한다.

### 4.4 AI 팀

목표:

- AI가 그럴듯한 말을 하는 것이 아니라, 데이터와 출처에 묶인 답변을 하게 만든다.
- VC가 봐도 AI/RAG/Agentic 확장성이 보이게 만든다.

필수 설계:

- AI service는 독립 서비스로 둔다.
- backend는 외부 API gateway 역할을 한다.
- RAG 대상:
  - daily summaries
  - stock events
  - OHLCV/chart signals
  - search results
  - learning terms
  - portfolio sandbox
  - 추후 DART/news/disclosure
- 답변 구조:
  - 한 줄 결론
  - 근거
  - 반대 신호
  - 리스크
  - 초보자 설명
  - 출처
  - 신뢰도
  - 데이터 기준일
  - 한계

Agentic 역할:

- Data Collector
- Event Detector
- Evidence Ranker
- Beginner Explainer
- Risk Reviewer
- Response Composer

투자 표현 원칙:

- `무조건 사라`, `지금 팔아라`, `반드시 오른다`, `수익 보장` 금지
- 대신 `매수 검토 구간`, `분할매수 검토 구간`, `관망`, `매도 검토 구간`, `리스크 관리 구간`으로 표현
- 차트 기반 조건, 근거, 반대 신호, 신뢰도, 기준일, 초보자 설명을 함께 제공

### 4.5 DevOps 팀

목표:

- 개발자가 빠르게 반복하고, 최종적으로는 Docker 기반으로 안정 실행 가능한 상태를 만든다.

필수 구현/점검:

- Docker Compose 전체 실행 유지
- `.env.example` 갱신
- secret commit 방지
- healthcheck 유지
- frontend/backend/marketdata/ai-service 실행 경로 문서화
- 배포 구조 검토
- CI/CD 또는 최소 release checklist 정리
- 로그와 장애 확인 방법 문서화

절대 규칙:

- `.env`, `.env.*`, API key, DB password, Discord webhook, OpenAI key, admin key는 절대 git에 올리지 않는다.
- 무거운 테스트를 매 루프마다 돌리지 않는다.
- 무거운 테스트는 마일스톤 또는 최종 보고 직전에 실행한다.
