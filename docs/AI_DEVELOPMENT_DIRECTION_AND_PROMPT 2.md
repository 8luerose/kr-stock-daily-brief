# AI 개발 방향 및 인수인계 프롬프트

작성일: 2026-05-03

## 1. 제품 방향 결정

현재 프로젝트는 `한국 주식 일간 브리프 운영 도구`에 가깝다. 이미 날짜별 시장 요약, KOSPI/KOSDAQ 상승/하락 TOP3, 최다 언급 종목, 휴장일 처리, 검증 링크, Docker Compose 실행 구조가 있다.

앞으로의 방향은 기존 기능을 버리지 않고, 이를 `주식 초보자를 위한 한국 주식 AI 리서치/학습 보조 앱`으로 확장하는 것이다.

추천 포지셔닝:

> "오늘 시장에서 어떤 종목이 왜 움직였는지, 초보자도 이해할 수 있는 언어로 설명하고, 관심 종목과 포트폴리오를 근거 중심으로 비교해주는 AI 주식 리서치 코파일럿"

중요한 제품 원칙:

- "무조건 사라/팔아라" 형태의 단정형 투자 지시는 피한다.
- 사용자의 투자 성향, 보유 종목, 금액, 매수/매도 시점을 직접 지시하는 기능은 법적/규제 검토 전에는 만들지 않는다.
- 대신 `관심 후보`, `판단 근거`, `리스크`, `초보자 설명`, `체크리스트`, `시나리오`, `학습형 Q&A` 중심으로 설계한다.
- 모든 AI 답변은 출처, 신뢰도, 한계, 데이터 기준일을 함께 보여준다.

참고:

- 국가법령정보센터 자본시장법 제6조는 투자자문업을 금융투자상품의 가치 또는 종목, 취득/처분, 가격, 시기 등에 관한 투자판단 자문으로 정의한다. https://www.law.go.kr/
- 국가법령정보센터 자본시장법 제18조는 투자자문업/투자일임업 등록 요건을 둔다. https://www.law.go.kr/

## 2. 대상 사용자

1차 사용자:

- 한국 주식을 막 시작한 개인 투자자
- 용어, 차트, 공시, 뉴스, 등락률의 의미를 잘 모르는 사람
- "오늘 왜 이 종목이 올랐지?", "이 버튼/지표가 무슨 뜻이지?", "이 종목을 살펴봐도 되는 이유와 위험은 뭐지?"를 알고 싶은 사람

2차 사용자:

- 매일 장 마감 후 시장을 빠르게 복기하고 싶은 개인/소규모 팀
- 특정 날짜의 시장 흐름을 기록하고 비교하려는 사용자

제외할 사용자:

- 초단타/실시간 매매 신호를 원하는 사용자
- 확정 수익률, 자동매매, 직접 매수/매도 지시를 기대하는 사용자

## 3. 핵심 사용자 경험

### 3.1 오늘의 시장 브리프

첫 화면은 달력 중심 운영 화면이 아니라, 초보자가 바로 이해하는 `오늘의 브리프`여야 한다.

구성:

- 오늘 시장 한 줄 요약
- KOSPI/KOSDAQ 상승/하락 TOP3
- 최다 언급 TOP3
- 왜 움직였는지에 대한 쉬운 설명
- 데이터 기준일, 출처, 신뢰도
- "AI에게 물어보기" 진입점

### 3.2 종목 상세 리서치

종목을 클릭하면 초보자용 종목 상세 화면으로 이동한다.

구성:

- 종목 기본 정보
- 최근 가격/거래량/등락률
- 일봉/주봉/월봉 차트
- 차트 위 이벤트 마커
- 급등/급락 이유 툴팁
- 관련 뉴스/공시/토론방 언급량
- AI 요약: "초보자에게 중요한 점 3가지"
- 리스크: "주의해야 할 점 3가지"

### 3.3 차트 이벤트 설명

사용자가 말한 핵심 기능이다.

필수 기능:

- 일봉/주봉/월봉 전환
- 호버 가능한 캔들/라인 차트
- 급등/급락 지점 자동 감지
- 차트 위 마커 표시
- 마커 호버 시 "왜 올랐는지/내렸는지" 툴팁 표시
- 툴팁에는 뉴스, 공시, 거래량, 토론방 언급량, 시장 전체 흐름 등 근거 링크 포함

추천 구현:

- Frontend: `lightweight-charts` 또는 검증된 캔들차트 라이브러리
- Backend/marketdata: OHLCV + event detection API
- AI service: 이벤트 원인 설명 생성

### 3.4 AI 질문/답변

앱 전체에 AI 패널을 둔다.

예시 질문:

- "이 종목 왜 올랐어?"
- "PER이 뭐야?"
- "이 뉴스가 주가에 왜 중요해?"
- "초보자 관점에서 이 종목을 볼 때 무엇을 확인해야 해?"
- "내 관심 종목 5개 중 변동성이 큰 종목은 뭐야?"

답변 원칙:

- 초보자 언어로 설명
- 어려운 용어는 한 문장으로 풀어쓰기
- 근거 링크와 기준일 표시
- 확실하지 않으면 불확실하다고 말하기
- 매수/매도 단정 대신 판단 체크리스트 제공

### 3.5 포트폴리오 샌드박스

초기에는 실계좌 연동 없이 가상 포트폴리오로 만든다.

구성:

- 관심 종목 담기
- 가상 비중 입력
- 섹터/변동성/상승·하락 민감도 표시
- "초보자에게 위험할 수 있는 이유" 설명
- "다음에 확인할 체크리스트" 제공

## 4. AI/RAG/Agentic/Harness 설계 방향

### 4.1 RAG

RAG의 목적은 "AI가 그럴듯한 말을 하는 것"이 아니라, 데이터와 출처에 묶인 답변을 하게 만드는 것이다.

수집/검색 대상:

- 저장된 일간 브리프
- KRX/OHLCV 기반 가격 데이터
- 네이버 금융 종목 페이지/토론방 언급량
- DART 공시
- 뉴스/리포트/공식 자료는 추후 소스 정책 확정 후 추가
- 앱 내부 용어 사전

AI 답변 구조:

- 결론
- 근거
- 반대 신호/주의점
- 초보자 설명
- 출처 링크
- 신뢰도

### 4.2 Agentic

한 번의 질문에 여러 에이전트 역할이 순차/병렬로 작동하게 설계한다.

추천 역할:

- Data Collector: 가격, 거래량, 브리프, 공시, 뉴스 후보 수집
- Event Detector: 급등/급락/거래량 급증/언급량 증가 탐지
- Evidence Ranker: 근거의 신뢰도와 관련도 평가
- Beginner Explainer: 초보자 언어로 설명
- Risk Reviewer: 과장, 단정, 매수/매도 지시, 근거 부족 표현 제거
- Response Composer: 최종 응답 생성

### 4.3 Harness Engineering

모든 AI 답변은 아래 내부 절차를 거친다.

1. 사용자 질문 의도 분류
2. 필요한 데이터 조회
3. 근거 기반 초안 작성
4. 자기 검토
5. 위험 표현 제거
6. 최종 답변 생성

금지:

- 출처 없는 확정 표현
- "반드시 오른다", "지금 사라", "지금 팔아라"
- 과거 수익률을 미래 수익처럼 표현
- 데이터 기준일 누락

허용:

- "관심 후보로 살펴볼 수 있음"
- "이런 근거 때문에 시장 관심이 커졌을 가능성이 있음"
- "초보자는 아래 3가지를 확인해야 함"
- "투자 결정 전 본인 기준과 손실 가능성을 확인해야 함"

## 5. 추천 기술 아키텍처

기존 구조는 유지한다.

- `frontend`: React + Vite. 초보자용 UX, 차트, AI 패널 담당.
- `backend`: Spring Boot. 인증, DB 저장, REST API, 스케줄러, Discord/운영 담당.
- `marketdata-python`: FastAPI + pykrx + Naver. 시장 데이터 계산 담당.
- `mysql`: 기존 요약/종목/이벤트/포트폴리오 저장.

추가 추천:

- `ai-service`: FastAPI 기반 LLM/RAG 서비스.
- `vector store`: Qdrant 또는 pgvector. Docker Compose로 함께 실행.
- `news/disclosure ingestion`: 별도 워커 또는 marketdata-python 내부 모듈로 시작.

Backend는 외부에 공개되는 단일 API 게이트웨이 역할을 하고, `ai-service`와 `marketdata-python`은 내부 서비스로 둔다.

- 휴장일/비거래일 정책 확정

- 종목 TOP3 카드 클릭 → 종목 상세로 연결
- 모든 데이터에 기준일/출처/신뢰도 표시

### Phase 2. 차트/이벤트 기능

- 일봉/주봉/월봉 차트 구현

- 차트 마커와 호버 툴팁 구현
- 이벤트별 근거 링크 저장

### Phase 3. AI/RAG 기능

- AI 서비스 추가

- `/api/ai/chat` 또는 `/api/research/ask` 추가
- 답변에 출처/신뢰도/한계 포함
- Risk Reviewer 단계 추가

### Phase 4. 포트폴리오 샌드박스

- 관심 종목 저장

- `.env.example` 갱신

- 차트 라이브러리는 직접 구현하지 말고 lightweight-charts 등 검증된 라이브러리를 우선 사용한다.

Phase 3. AI/RAG 서비스
- ai-service(FastAPI)를 추가하는 방향을 우선 검토한다.
- backend는 외부 API 게이트웨이, ai-service는 내부 AI/RAG 담당으로 둔다.
- Docker Compose에 ai-service와 vector store(Qdrant 또는 pgvector)를 추가한다.
- RAG 대상:
  - 저장된 daily_summaries
  - 종목 이벤트
  - OHLCV 기반 지표
  - 네이버 토론방 언급량
  - DART 공시/뉴스는 소스 정책 확정 후 추가
  - 초보자용 용어 사전
- API 예:
  POST /api/ai/chat
  Request: { "question": "...", "context": { "date": "...", "ticker": "...", "portfolioId": "..." } }
  Response: { "answer": "...", "confidence": "...", "sources": [...], "limitations": [...] }

AI 답변 Harness:
1. 질문 의도 분류
2. 필요한 데이터 검색
3. 근거 기반 초안 생성
4. 리스크/규제/과장 표현 검토
5. 초보자 친화 문장으로 재작성
6. 출처/신뢰도/한계 포함하여 응답

Phase 4. 포트폴리오 샌드박스
- 실계좌 연동 없이 관심 종목/가상 비중 기반으로 시작한다.
- 사용자는 종목과 비중을 저장할 수 있다.
- 시스템은 섹터 집중도, 변동성, 최근 이벤트, 하락 리스크를 초보자 언어로 설명한다.
- "매수/매도 지시"가 아니라 "확인할 체크리스트"를 제공한다.

디자인 요구사항:
- Toss처럼 단정하고 가볍고 신뢰감 있는 UI
- 첫 화면은 마케팅 랜딩이 아니라 실제 사용 가능한 대시보드
- 카드 남발 금지. 정보 밀도는 높되 읽기 쉽게 정리
- 색은 파랑/흰색/회색 중심으로 하되 상승/하락은 명확히 구분
- 모바일에서 텍스트와 버튼이 겹치지 않게 검증
- 초보자가 모르는 용어에는 즉시 설명 또는 툴팁 제공

데이터 정확도 규칙:

## 8. 이번 전환의 핵심 결론

이 프로젝트의 강점은 이미 `날짜별 시장 데이터 수집/저장/검증/표시` 기반이 있다는 점이다. 따라서 완전히 새 AI 앱을 만드는 것보다, 현재 브리프 시스템을 데이터 기반으로 삼아 초보자용 설명, 차트 이벤트, RAG Q&A, 포트폴리오 샌드박스를 단계적으로 얹는 방향이 가장 효율적이다.

가장 먼저 바꿀 것은 기술이 아니라 첫 화면의 목적이다. 사용자가 앱을 열었을 때 "관리자가 요약을 생성하는 도구"가 아니라 "오늘 주식시장을 이해하게 도와주는 화면"으로 느껴져야 한다.
