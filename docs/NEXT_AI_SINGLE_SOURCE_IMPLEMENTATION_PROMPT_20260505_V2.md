# Next AI Single Source Implementation Prompt 20260505 V2

아래 내용을 다른 AI에게 그대로 전달한다.

이 문서는 기존 여러 MD 문서의 요구사항을 하나로 종합한 `단일 실행 프롬프트`다.
다른 AI에게 모든 MD 문서를 다시 읽으라고 시키지 않는다.
이 문서 하나를 기준으로 개발, 반복 개선, 마지막 검증, 보고까지 수행한다.

## 0. 이번 작업의 진짜 목표

이번 목표는 `프론트 전면 재설계와 재구현`이다.

테스트, 감사, 점수표, Docker, E2E, 전체 API 검증에 시간을 쓰는 것이 1차 목표가 아니다.
먼저 사용자가 눈으로 봤을 때 "완전히 새 제품이 됐다"고 느낄 정도로 프론트를 바꿔라.

가장 중요한 해석:

- 백엔드 기능은 삭제하지 마라.
- 기존 API, DB, marketdata, ai-service, Docker 구조는 보존해라.
- 하지만 프론트는 지우고 다시 만들어도 된다.
- 기존 프론트 UI, 레이아웃, CSS, 컴포넌트 구조에 얽매이지 마라.
- 기존 기능은 프론트에서 삭제하는 것이 아니라, 일반 사용자 화면과 관리자/기록 영역으로 재배치한다.

## 1. 역할

너는 `kr-stock-daily-brief` 프로젝트의 프론트 전면 재구현을 맡은 시니어 제품 개발 AI다.

너는 한 명이지만 아래 역할을 모두 수행한다고 가정한다.

- Toss에서 10년차 이상, 연봉 1억 이상 받는 수준의 프론트엔드 개발자
- 사용자가 첫 화면에서 바로 가치를 이해하게 만드는 제품 디자이너
- 차트 기반 투자 학습 UX를 설계하는 핀테크 UX 전문가
- AI 기능을 화면에서 명확하게 보이게 만드는 AI 제품 엔지니어
- 기존 백엔드 기능을 깨지 않도록 확인하는 풀스택 엔지니어

## 5. 절대 규칙

- 테스트부터 돌리지 마라.
- E2E, Docker rebuild, make quality, 전체 API 테스트, 긴 smoke test에 초반 시간을 쓰지 마라.
- 먼저 프론트 화면을 실제로 바꿔라.
- 기존 프론트 UI를 살리려고 하지 마라.
- 기존 프론트 코드는 필요하면 크게 삭제하거나 재작성해도 된다.
- 단, 백엔드 기능과 기존 API 동작은 삭제하지 마라.
- 기존 브리프, 달력, 생성, 백필, 보관, 검증 기능은 프론트에서 관리자/기록 영역으로 옮겨 보존해라.
- `.env`, `.env.*`, API key, DB password, Discord webhook, OpenAI key, admin key 같은 secret은 절대 commit하지 마라.
- 기존 사용자의 변경을 함부로 되돌리지 마라.
- 실패하거나 확인하지 못한 것은 반드시 실패/미확인이라고 보고해라.
- 커밋 전에는 반드시 `git log`를 확인하고, 기존 프로젝트의 커밋 메시지 흐름과 컨벤션을 따른다.

## 6. 현재 반드시 해결해야 할 문제

- 화면이 기존과 크게 달라 보이지 않는다.
- 버튼이 너무 많고 복잡하다.
- Toss처럼 깔끔하고 고급스럽지 않다.
- 애니메이션과 depth가 약하다.
- 첫 화면에서 이 서비스가 무엇이고 왜 좋은지 바로 알기 어렵다.
- 원하는 산업, 테마, 기업, 종목을 검색하는 경험이 부족하다.
- 그래프를 기준으로 실제 공부되는 느낌이 약하다.
- AI 기능이 어디인지, 왜 유용한지, 왜 확장 가능한지 잘 보이지 않는다.
- 학습 탭의 주식 용어 설명이 적고 얕다.
- PC와 모바일에서 잘림, 겹침, 화면 끝 붙음 문제가 없어야 한다.

## 8. 프론트 구현 지시

### 8.1 구조

기존 `App.jsx`, `AppSections.jsx`, `styles.css` 구조에 얽매이지 마라.

필요하면 아래를 새로 만든다.

- Command center
- Search

- AI insight panel
- Learning panel

목표:

- 재사용성
- 가독성
- PC/mobile 반응형 안정성
- UI 확장성

### 8.2 첫 화면

첫 화면에는 아래만 선명하게 둔다.

- 통합 검색
- 메인 차트
- AI 핵심 인사이트
- 관심 후보 3개 이하
- 학습 진입점
- 데이터 기준일
- 신뢰도

첫 화면에서 숨기거나 관리자/기록 영역으로 이동할 것:

- 긴 브리프 전문
- 검증 상세
- 수집 노트
- 생성/백필/보관
- 긴 TOP 리스트
- 외부 링크 반복
- 관리자 버튼

### 8.3 UI/UX 감성

- Toss처럼 단정하고 신뢰감 있어야 한다.
- 심플하지만 심심하지 않아야 한다.
- 버튼은 모바일 앱처럼 적고 명확해야 한다.
- 전면 primary action은 1~3개만 둔다.
- Pretendard를 한국어/영어 기본 폰트로 적용한다.
- 데이터, 차트, 검색, AI 결과가 주인공이 되게 한다.
- hover, focus, active, panel transition, tooltip, loading, empty, error state에 150~320ms 수준의 부드러운 micro interaction을 넣어라.
- 글래스모피즘은 과하지 않게 사용한다.

## 13. 백엔드 관련 지시

백엔드는 삭제하지 마라.

삭제하면 안 되는 기능:

- summaries API
- 날짜별 조회
- 최신 조회
- 통계
- 인사이트
- 생성
- 백필
- 보관
- 검증
- 기존 브리프/달력/운영 흐름

백엔드 수정은 아래 경우에만 한다.

- 프론트가 필요한 API adapter가 없을 때
- 검색/차트/AI 패널이 실제 데이터를 받지 못할 때
- 기존 API 계약과 프론트 요구가 맞지 않을 때
- 관리자/일반 사용자 기능 분리를 위해 routing 정리가 필요할 때

데이터 정확도 규칙:

- pykrx의 `get_market_price_change_by_ticker()` 등락률을 그대로 믿지 마라.
- 전일 종가 대비 등락률은 OHLCV 당일/전일 종가로 직접 계산해라.
- 거래량 0, 비정상 종목코드, 거래정지 의심 종목을 필터링해라.
- `0011T0` 같은 비6자리 숫자 코드는 제외해라.
- KOSPI/KOSDAQ 상승/하락 1위와 TOP3 첫 항목은 반드시 일치해야 한다.

## 19. Git 커밋 메시지 작성 규칙

커밋 전에는 반드시 최근 커밋을 확인한다.

확인 명령 예시:

```bash
git log --oneline -10
```

기존 프로젝트의 커밋 메시지 흐름을 확인한 뒤, 아래 형식을 따른다.

```text
[FEAT]: (1) 내용1 / (2) 내용2 / (3) 내용3
```

대괄호 안에는 커밋의 성격을 쓴다.

- `[FEAT]`: 새로운 기능을 만들었을 때
- `[REFACTOR]`: 기존 기능의 구조나 동작 방식을 변경했을 때
- `[CHORE]`: 자잘한 코드, 설정, 정리 작업을 했을 때
- `[DOCS]`: 새 문서를 작성하거나 기존 문서를 수정했을 때
- `[STYLE]`: 기능 변화 없이 코드 스타일, UI 스타일, 포맷을 정리했을 때

커밋 메시지는 어려운 말이나 줄임말을 피한다.
작성자가 아닌 다른 사람이 봐도 바로 이해할 수 있어야 한다.

커밋하기 전에는 해당 커밋에 적합한 설명 후보를 `(N)`개로 나열하고, 그 설명을 포함한 커밋 메시지 예시를 제시한다.
가능하면 사용자에게 선택지를 보여주고, 상황상 직접 커밋해야 한다면 가장 명확한 메시지를 선택한다.

설명 후보 예시:

```text
(1) 첫 화면을 검색, 차트, AI 인사이트 중심 구조로 다시 구성
(2) 기존 운영 기능을 일반 사용자 화면에서 관리자 영역으로 이동
(3) 모바일 화면에서 검색 결과와 차트 툴팁이 잘리지 않도록 레이아웃 수정
```

커밋 메시지 예시:

```text
[FEAT]: (1) 첫 화면을 검색, 차트, AI 인사이트 중심 구조로 다시 구성 / (2) 기존 운영 기능을 일반 사용자 화면에서 관리자 영역으로 이동 / (3) 모바일 화면에서 검색 결과와 차트 툴팁이 잘리지 않도록 레이아웃 수정
```

다른 예시:

```text
[REFACTOR]: (1) 프론트 화면 구조를 App shell, 검색 영역, 차트 작업 영역으로 분리 / (2) 공통 API 호출 로직을 재사용 가능한 client로 정리 / (3) 관리자 기능을 별도 패널에서 관리하도록 구조 변경
```

문서만 수정한 경우 예시:

```text
[DOCS]: (1) 다음 AI에게 전달할 단일 실행 프롬프트를 프론트 전면 재구현 중심으로 수정 / (2) 테스트는 마지막에만 실행하도록 작업 순서를 명확히 정리 / (3) 커밋 메시지 작성 규칙과 예시를 추가
```
